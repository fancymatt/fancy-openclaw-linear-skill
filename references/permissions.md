# Agent Permissions Guide

This guide explains how to provision Linear credentials for OpenClaw agents with the correct permissions. It covers both credential types and the specific scopes needed for this skill's commands.

## Credential Types

### Personal API Key (`lin_api_...`)

Generated by a human user in **Settings â†’ Account â†’ Security & access â†’ API**. Inherits that user's exact permissions.

**Best for:** Agents that act as a specific person (e.g. "Charles" acting as the CTO user). The agent does everything the human could do.

**Setup:**
1. Log in to Linear as the user
2. Settings â†’ Account â†’ Security & access â†’ API â†’ New token
3. Label it (e.g. `openclaw-agent-charles`)
4. Choose permissions (see below)
5. Optionally restrict to specific teams

### Developer Token / OAuth Application (`lin_oauth_...`)

Created through a Linear OAuth application. Has its own identity separate from any human user.

**Best for:** Agents that act as a bot or service (e.g. "Mckell" as an app-style agent that isn't tied to a specific human account).

**Setup:**
1. An admin creates an OAuth application at **Settings â†’ Administration â†’ API â†’ Applications**
2. Configure the redirect URI and scopes
3. Authorize the application with `actor=app` so resources are created as the application
4. The resulting access token is what the agent uses

**Note:** OAuth tokens expire (24-hour access tokens, with refresh tokens). Plan for renewal if using long-running agents.

## Recommended Permissions

### For agents that manage issues (most agents)

When creating the token, grant these scopes:

| Permission | Why needed | Commands that require it |
|---|---|---|
| **Read** | All query commands | `issue`, `my-issues`, `board`, `states`, `projects`, `teams`, `comments`, `review-queue`, `stalled`, `standup`, `notifications`, `urgent`, `children`, `relations`, `branch` |
| **Write** | Mutate issues | `status`, `assign`, `priority`, `create`, `comment`, `subtask`, `handoff`, `block`, `unblock`, `project-attach`, `milestone-attach`, `upload` |
| **Create issues** | Create new issues | `create`, `subtask` |
| **Create comments** | Add comments to issues | `comment`, `handoff` |

**Recommended setup:** Grant **all four** (Read, Write, Create issues, Create comments). There's no benefit to restricting a trusted agent â€” it just creates confusing failures when a command unexpectedly doesn't work.

### For agents that also manage projects and milestones

You also need **Admin** access or the token must belong to a workspace admin user. This is needed for:
- `milestone-create`
- `project-attach` (to some project configurations)

### For read-only agents (monitoring, dashboards)

Grant only **Read**. All query commands will work. Any mutation will return a permissions error, which is the correct behavior.

## Team Restrictions

When creating a personal API key, you can optionally restrict it to specific teams. This limits the agent to only seeing and mutating issues in those teams.

**For most agents:** Don't restrict. Agents like Charles, Astrid, etc. work across all teams.

**For team-specific agents:** Restrict to the relevant team. The agent will get errors when trying to access issues outside their team, which is correct.

## What Happens With Insufficient Permissions

| Symptom | Cause | Fix |
|---|---|---|
| `400 Bad Request` / `Entity not found: Team` | Passing team key instead of UUID to `team(id:)` | This is a bug, not permissions. Was fixed in AI-227. |
| `401 Unauthorized` | Token is invalid or revoked | Regenerate the token |
| `403 Forbidden` | Token lacks the required scope | Grant additional permissions in Linear settings |
| Empty results on `my-issues`, `board`, etc. | Token is restricted to teams that have no matching data | Check team restrictions, or assign issues to the agent |
| `viewer.assignedIssues` returns empty | The agent user has nothing assigned to them | This is correct behavior, not an error |

## Mckell Pilot Root Cause

The Mckell pilot (AI-224) showed 400 errors on `board AI`, `states AI`, and `my-issues AI`. Investigation (AI-230) revealed:

- **Not a permissions issue.** Mckell's OAuth token has full read/write access.
- **Root cause:** `board` and `states` commands were passing the team key string (`"AI"`) directly to Linear's `team(id:)` field, which expects a UUID. The `resolveTeamId` function added in AI-227 fixes this by resolving keys to UUIDs.
- **`my-issues`** was working correctly â€” it returned empty because Mckell had no issues assigned. The 400 reported during the pilot may have been from a different invocation or has been resolved by subsequent changes.

## Setup Checklist for a New Agent

1. Decide: personal API key or OAuth developer token?
2. Generate the token with **Read + Write + Create issues + Create comments** (all four scopes)
3. Store in `~/.openclaw/workspace-{agent}/.secrets/linear.env`
4. `npm run build && node dist/index.js auth check --human` â€” should show the agent's name
5. `node dist/index.js teams` â€” should list all teams
6. `node dist/index.js board AI` â€” should show the AI Systems board (verifies team key resolution works)
7. `node dist/index.js my-issues` â€” should return issues (or empty if none assigned)
